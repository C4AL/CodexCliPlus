# Cli Proxy API Desktop 已确认事项

以下条目为当前确认的产品边界与实现约束。

## 产品边界

- 产品唯一主名：`Cli Proxy API Desktop`
- 产品简称：`CPAD`
- 只支持 `Windows`
- 只支持 `中文`
- 不再提供其他平台与多语言选项
- 当前主仓必须保持为可独立构建的 CPAD 项目；旧项目标识仅允许出现在来源声明、兼容说明和基线记录中

## 主仓与外部资产边界

- CPAD 主仓只保留产品本体代码，不直接把旧仓整棵复制进来
- `CPA-UV`、官方主程序基线、官方管理中心基线、插件源仓都视为受控外部资产
- 受控外部资产可以被同步、比对、抽职责、迁能力，但不反向污染主仓结构
- 任何能力并入主仓前，都必须先明确职责归属、来源与许可证

## 安装与目录

- 默认安装路径：`C:\Users\<主用户>\Cli Proxy API Desktop\`
- 主程序 `exe` 直接放安装根目录
- 所有真实数据统一放安装目录内
- 允许在安装目录内建立 `data` 子目录存放 SQLite 和其他持久化数据
- 允许在安装目录内建立 `upstream` 子目录存放受控官方基线工作树
- 不使用 WSL 作为默认运行环境

## 运行形态

- 采用 `Windows Service + 用户态 Chromium 前端`
- 开机即启动
- 未登录也运行
- 桌面前端只负责交互与管理，不承担后台常驻职责
- 受控 CPA Runtime 的默认托管端口为 `2723`
- 旧默认端口仅在当前配置仍为历史默认值时自动迁移，不覆盖用户自定义端口

## 官方基线策略

- 更新中心必须同时跟踪 `router-for-me/CLIProxyAPI` 与 `router-for-me/Cli-Proxy-API-Management-Center`
- 官方双基线必须允许同步到安装目录内的受控工作树
- “同步官方基线”只代表基线更新，不代表已经完成 `CPA-UV` 覆盖合并
- 官方管理中心基线只作为受控来源，不等于允许把其原始结构整仓并入 CPAD

## Codex 接管

- 终端里 `codex` 的入口由桌面版接管
- 使用机器级 `codex` shim 转发到安装目录内的官方 Codex CLI 私有运行时
- 桌面端可切换官方模式 / CPA 模式
- 切换仅影响之后新启动的 Codex 会话
- 切换时需要明确提示用户该行为
- 不鼓励也不支持用户直接通过 `npm` 升级受控 Codex CLI

## 更新策略

- 统一更新入口
- 桌面端统一管理前端、后端、模式控制和更新流程
- Codex CLI 更新遵循官方更新源，但是否放行由桌面端兼容策略决定
- 插件市场有独立插件更新源，但入口统一在桌面端内
- 更新中心要同时反映官方基线、本地覆盖源仓和受控运行时状态

## 同步策略

- 运行时、配置、插件索引、桥接目录和官方基线工作树相关的变更同步采用点对点方式
- 每次同步只允许一个明确源与一个明确目标
- 不引入多点广播式自动合并，也不引入新的中心化多端同步层
- 同步不得制造第二真实数据源，安装目录仍是唯一真实数据源

## 数据与迁移

- 默认全新安装
- 不做首次导入向导
- 不自动导入旧私密配置、凭证和隐私内容
- 真实数据迁移到安装目录后，旧路径只保留最小桥接
- 如后续需要导入旧环境，放到设置页中手动触发，且默认只导入非敏感内容

## 插件市场

- 只支持自有插件集合
- 插件市场内置在桌面端
- 暂不扩展泛化第三方插件生态

## 依赖约束

- 只纳入 CPAD 与 Codex 主链路所需的必要依赖和前置
- 不主动引入无关第三方组件
- 对旧桥接、旧脚本和一次性迁移逻辑保持克制，能删除的要删除，不能长期背在主链路上

## 旧项目关系

- 旧仓只作为来源、基线和许可证记录保留
- 有价值能力按职责拆解后再并入当前主仓
- `Cli Proxy API Desktop` 是当前唯一主仓库
