# Cli Proxy API Desktop 完整开发计划

## 1. 项目定位

Cli Proxy API Desktop 的目标不是继续维护一个“带 WebUI 的 CPA 分支”，而是把以下能力正式产品化：

- CPA 后端能力
- Codex 本地模式 / 官方模式切换能力
- 受控 `codex` 入口接管
- 统一更新中心
- 安装目录内持久化统计与状态数据库
- 自有插件市场
- Windows 桌面入口与后台常驻能力

最终产品形态：

- 一个主仓库
- 一个桌面品牌
- 一个安装器
- 一个更新入口
- 一个安装目录

## 2. 架构总览

### 2.1 进程模型

- `Cli Proxy API Desktop.exe`
  - Electron/Chromium 桌面前端
  - 用户态运行
  - 负责 UI、设置、状态查看、更新触发、插件市场、日志查看

- `Cli Proxy API Desktop Service`
  - Windows Service
  - 负责开机常驻、未登录运行、后台调度、模式状态、数据库、更新任务、后端守护

- `CPA Runtime`
  - 受桌面端管理的后端运行时
  - 由服务负责拉起、监控、重启、版本切换

- `Codex Shim`
  - 机器 PATH 前置的薄代理入口
  - 用户在终端输入 `codex` 时，先进入该 shim
  - shim 读取当前模式，再委托安装目录内的官方 Codex CLI 私有运行时

### 2.2 安装目录

统一安装根目录：

`C:\Users\<主用户>\Cli Proxy API Desktop\`

推荐目录布局：

- `Cli Proxy API Desktop.exe`
- `Cli Proxy API Desktop Service.exe`
- `data\app.db`
- `data\codex\`
- `data\cpa\`
- `runtime\codex\`
- `runtime\cpa\`
- `plugins\`
- `logs\`
- `tmp\`

说明：

- 主程序直接放根目录，便于查找
- 真实数据统一放安装目录内
- 不依赖 `%USERPROFILE%\.codex` 或 `%USERPROFILE%\.cli-proxy-api` 作为真实数据源

### 2.3 兼容桥接

为兼容官方 Codex CLI 和旧插件的默认查找路径，保留最小桥接：

- `%USERPROFILE%\.codex`
- `%USERPROFILE%\.cli-proxy-api`

桥接原则：

- 只做最小跳转或兼容文件
- 真实数据始终在安装目录
- 卸载时一并清理桥接残留

### 2.4 同步原则

- 涉及运行时、配置、插件索引和桥接目录的变更同步，统一采用点对点方式
- 每次同步只允许一个明确源与一个明确目标
- 不做多点并写、广播式自动合并或新的中心化同步层
- 安装目录始终是唯一真实数据源，同步只负责分发、落地或兼容

## 3. 产品目标拆解

### 3.1 安装器

首版安装器需要完成：

- 选择主用户
- 安装到 `C:\Users\<主用户>\Cli Proxy API Desktop\`
- 下载 CPA Runtime
- 下载官方 Codex CLI Windows 运行时
- 写入机器级 `codex` shim
- 注册 Windows Service
- 创建桌面快捷方式
- 初始化安装目录数据库与配置骨架

### 3.2 后台服务

后台服务负责：

- 开机自启
- 未登录时运行
- 守护 CPA Runtime
- 管理模式状态
- 维护 SQLite 数据库
- 执行更新检查和下载
- 维护插件索引和安装状态

### 3.3 桌面前端

桌面前端的首版核心页面：

- 总览
- 运行状态
- 模式切换
- Codex 设置
- CPA 设置
- 插件市场
- 更新中心
- 日志与诊断

### 3.4 Codex 管理

Codex 相关目标：

- 桌面端切换官方模式 / CPA 模式
- 切换时提示“仅影响之后新启动的 Codex 会话”
- 屏蔽直接通过 `npm` 升级带来的失控更新
- 统一由桌面更新中心放行 Codex CLI 版本

### 3.5 插件市场

第一阶段只支持自有插件集合。

来源范围：

- 现有自有 Codex 插件
- 后续迁入的新项目内置插件市场清单

不做：

- 泛化第三方插件市场
- 与主链路无关的插件分发体系

### 3.6 统计与持久化

统一使用 SQLite 持久化：

- 运行状态
- 请求统计
- 额度聚合
- 更新记录
- 插件安装记录
- 模式切换历史

目标：

- 后端重启后数据不丢失
- 安装目录内一站式管理

## 4. 迁移计划

### 4.1 第一阶段：建库与冻结

- 创建新仓 `Cli-Proxy-API-Desktop`
- 写入 README、AGENTS、开发计划、已确认事项
- 建立待整合资产清单
- 归档旧仓，确立新仓为唯一主方向

### 4.2 第二阶段：基础设施

- 确定 Electron 技术栈
- 建立 Windows Service 宿主
- 建立安装目录约定
- 建立 SQLite 数据层
- 建立 Codex shim

### 4.3 第三阶段：后端接管

- 把现有 CPA-UV 运行时纳入桌面端受控运行
- 抽离旧脚本中的模式切换逻辑
- 抽离更新逻辑
- 抽离管理中心与前端状态数据接口
- 完成 CPAD 重建收口，清理旧标识文本、旧壳层语言与旧工程残留，确保当前主仓已经是结构干净、可独立构建的 CPAD 项目
- 在第三阶段收尾时合入官方最新代码仓 `router-for-me/CLIProxyAPI` 作为更新基线，并确保由 CPA-UV 覆盖官方基线内容、补齐兼容层，而不是回退或丢失 CPA-UV 已有能力

### 4.4 第四阶段：插件市场

- 建立内置市场清单格式
- 接管现有自有插件
- 提供安装、更新、启用、禁用和诊断入口

### 4.5 第五阶段：发布

- 打包 Windows 安装器
- 验证全新安装、升级安装、卸载流程
- 验证 Codex 模式切换
- 验证更新中心
- 验证服务常驻和数据库持久化

## 6. 风险与约束

- 未登录也运行必须依赖 Windows Service，不能退化成纯用户态方案。
- 统一更新入口不等于无限追随上游最新版本，特别是 Codex CLI 必须有兼容性放行策略。
- 单目录安装与全机安装并存时，需要提前处理 ACL 和服务账户访问权限。
- 桥接旧路径的实现必须最小化，否则会重新引入双份状态源。
- 点对点同步必须严格保持单源单目标，否则会在更新、桥接和迁移阶段放大状态分叉。
- 不做首次导入向导意味着首版体验更干净，但旧环境迁移功能需在后续设置页中补充。

## 7. 当前结论

从当前本机现状出发，Cli Proxy API Desktop 是可落地的。关键原因不是要重写一切，而是已有这些可复用资产：

- 运行中的 CPA-UV 能力
- 已脚本化的 Codex 本地接管逻辑
- 现有自有插件集合与市场索引
- 本机已经验证过的更新、切换、桥接和状态展示需求

下一阶段工作的核心，不是再造一个新的 CPA 分支，而是把这些零散能力做成一个真正的 Windows 桌面产品。
