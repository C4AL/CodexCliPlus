# Cli Proxy API Desktop 完整开发计划

## 1. 项目定位

Cli Proxy API Desktop 的目标不是继续维护一个“带 WebUI 的 CPA 分支”，而是把以下能力正式产品化：

- CPA 后端能力
- Codex 本地模式 / 官方模式切换能力
- 受控 `codex` 入口接管
- 统一更新中心
- 安装目录内持久化统计与状态数据库
- 自有插件市场
- Windows 桌面入口与后台常驻能力

最终产品形态：

- 一个主仓库
- 一个桌面品牌
- 一个安装器
- 一个更新入口
- 一个安装目录

## 3. 架构总览

### 3.1 进程模型

- `Cli Proxy API Desktop.exe`
  - Electron/Chromium 桌面前端
  - 用户态运行
  - 负责 UI、设置、状态查看、更新触发、插件市场、日志查看

- `Cli Proxy API Desktop Service`
  - Windows Service
  - 负责开机常驻、未登录运行、后台调度、模式状态、数据库、更新任务、后端守护

- `CPA Runtime`
  - 受桌面端管理的后端运行时
  - 由服务负责拉起、监控、重启、版本切换

- `Codex Shim`
  - 机器 PATH 前置的薄代理入口
  - 用户在终端输入 `codex` 时，先进入该 shim
  - shim 读取当前模式，再委托安装目录内的官方 Codex CLI 私有运行时

### 3.2 安装目录

统一安装根目录：

`C:\Users\<主用户>\Cli Proxy API Desktop\`

当前目录布局约定：

- `Cli Proxy API Desktop.exe`
- `Cli Proxy API Desktop Service.exe`
- `data\app.db`
- `data\codex\`
- `data\cpa\`
- `runtime\codex\`
- `runtime\cpa\`
- `plugins\`
- `logs\`
- `tmp\`
- `upstream\CLIProxyAPI\`
- `upstream\Cli-Proxy-API-Management-Center\`

说明：

- 主程序直接放根目录，便于查找
- 真实数据统一放安装目录内
- 官方双基线工作树也放在安装目录内的 `upstream\` 下受控管理
- 不依赖 `%USERPROFILE%\.codex` 或 `%USERPROFILE%\.cli-proxy-api` 作为真实数据源

### 3.3 受控外部资产

当前主线涉及的受控外部资产：

- 官方主程序基线：`router-for-me/CLIProxyAPI`
- 官方管理中心基线：`router-for-me/Cli-Proxy-API-Management-Center`
- 本地 CPA-UV 源仓：`C:\Users\Reol\workspace\CPA-UV-publish`
- 本地插件源仓：`C:\Users\Reol\workspace\omni-bot-plugins-oss`

原则：

- 主仓只保留 CPAD 自己的产品代码
- 官方基线、CPA-UV 和插件源都保留为外部受控资产
- 任何同步操作都必须遵守点对点、单源单目标

### 3.4 兼容桥接

为兼容官方 Codex CLI 和旧插件的默认查找路径，保留最小桥接：

- `%USERPROFILE%\.codex`
- `%USERPROFILE%\.cli-proxy-api`

桥接原则：

- 只做最小跳转或兼容文件
- 真实数据始终在安装目录
- 卸载时一并清理桥接残留

## 6. 风险与约束

- 未登录也运行必须依赖 Windows Service，不能退化成纯用户态方案
- 统一更新入口不等于无限追随上游最新版本，特别是 Codex CLI 必须有兼容性放行策略
- 单目录安装与全机安装并存时，需要提前处理 ACL 和服务账户访问权限
- 桥接旧路径的实现必须最小化，否则会重新引入双份状态源
- 点对点同步必须严格保持单源单目标，否则会在更新、桥接和迁移阶段放大状态分叉
- 管理中心能力不能机械复制原仓结构，必须按 CPAD 产品边界重新拆职责
- 官方双基线只是基线，不等于已经完成 `CPA-UV` 覆盖合并
- 不做首次导入向导意味着首版体验更干净，但旧环境迁移功能需在后续设置页中补充
